部署之前先核验环境，环境不满足，后面一定出问题

**很多部署失败**，不是因为软件不会装，而是因为前面没检查清楚：内存不够、端口被占用、权限不够、网络不通、系统版本不匹配、磁盘空间不足。部署人员的专业性，很大一部分就体现在环境核验

## 一，什么是环境核验
环境核验就是在正式安装部署前，**检查目标机器和运行条件是否满足系统要求**
这台电脑/服务器，能不能跑这个系统？
缺什么？
风险在哪里？
需要提前准备什么？

例子：**你要部署本地 AI 服务，就不能上来就安装。你要先问**：
电脑是什么系统？
内存多大？
显卡有没有？
磁盘够不够？
网络能不能下载模型？
端口有没有被占用？
用户有没有管理员权限？
有没有 Docker / Python / CUDA 等依赖？

## 二，环境核验的六大类

硬件环境
系统环境
网络环境
权限环境
依赖环境
业务环境
#### 1.硬件环境
检查机器性能够不够，常见的检查项
CPU 型号和核心数
内存大小
显卡型号
显存大小
磁盘容量
磁盘剩余空间
是否是 SSD
服务器是否稳定供电
**部署AI大模型时**，最重要的是：内存，显存，磁盘空间

#### 2.系统环境
**检查操作系统和基础环境**。常见的检查项：
Windows / Linux / macOS
系统版本
系统架构，x64 还是 ARM
是否支持目标软件
是否需要 WSL
是否需要虚拟化
是否有系统更新限制
例如：
Ollama 支持 Windows、macOS、Linux
一些 GPU 加速能力可能和系统、驱动、CUDA 有关
Odoo 部署常见环境是 Linux 或 Docker

#### 3.网络环境
**检查能不能下载、访问、连接**，常见检查项
能不能访问官网
能不能下载模型或镜像
内网能不能访问服务
端口是否开放
防火墙是否拦截
代理是否配置
DNS 是否正常
**主要熟悉：端口，防火墙**
端口可以理解为服务对外开门的门牌号
例如：Open WebUI 默认可能使用 3000 或 8080 之类端口
API 服务可能使用 8000
网页服务常见 80 / 443
端口被占用，服务就会失效

#### 4.权限环境
**检查你有没有资格做操作**，常见检查项
是否有管理员权限
是否能安装软件
是否能打开终端
是否能启动服务
是否能修改配置文件
是否能开放端口
是否能访问目标目录
**很多公司电脑会限制权限**。你没有管理员权限，可能连安装、改配置、开服务都做不了

#### 5.依赖环境
**检查目标系统需要的基础组件是否具备**，常见依赖
Python
Node.js
Docker
Git
CUDA
显卡驱动
数据库
浏览器
运行库
包管理工具
**例如：**
RAG 应用可能需要 Python、向量数据库、Embedding 模型
Open WebUI 常见部署方式可能需要 Docker
Odoo 可能需要 PostgreSQL

#### 6.业务环境
**实施岗位非常看重。**
你要检查：
谁使用系统？
有多少用户？
核心业务流程是什么？
旧数据在哪里？
是否需要迁移数据？
是否需要账号权限分级？
上线时间有没有限制？
失败后能不能回滚？
**部署不是只看机器，还要看业务能不能跑通**

### 三，环境的专业表达
**我在部署前会先做环境核验**，主要从硬件、系统、网络、权限、依赖和业务六个维度检查。这样可以提前发现内存不足、端口占用、权限缺失、依赖未安装、网络不通等风险，避免部署过程中反复返工。

### 四，以本地AI服务为例
部署 `Ollama + Open WebUI`，你的**环境核验表可以这样写：**
硬件：
CPU 是否可用
内存是否至少 16GB
磁盘剩余空间是否大于 20GB
是否有独立显卡
显存大小是多少

系统：
操作系统是否支持 Ollama
系统是 Windows / macOS / Linux
系统架构是否兼容
是否开启虚拟化或 WSL

网络：
是否能访问 Ollama 官网
是否能下载模型
是否能访问 Open WebUI 镜像或安装源
本机端口是否可用
防火墙是否拦截访问

权限：
是否有管理员权限
是否能安装软件
是否能启动后台服务
是否能修改环境变量
是否能访问安装目录

依赖：
是否安装 Docker
是否安装 Git
是否需要 Python
显卡驱动是否正常
是否需要 CUDA

业务：
谁使用这个 AI 服务
通过网页访问还是 API 访问
是否需要账号登录
是否需要保存聊天记录
是否只本机使用还是局域网使用

### 五，部署人员怎么排查环境问题
1. 服务启动失败：看端口是否被占用、权限是否不足
2. 模型下载失败：看网络、代理、磁盘空间
3. 网页打不开：看服务是否启动、防火墙、访问地址是否正确
4. 运行很慢：看内存、显存、CPU 占用
5. 功能异常：看日志、配置文件、依赖版本
**排查顺序记：**
先看服务是否启动
再看端口是否监听
再看网络是否能访问
再看日志报错
最后看配置和依赖

# 第二课作业：本地 AI 服务部署环境核验表

## 1. 部署目标
我要部署一个本地 AI 服务，供公司内部岗位使用，例如财务、采购、业务等人员。普通用户可以通过可视化网页访问，技术人员可以通过 API 或命令行进行测试和调用。部署前需要确认使用对象、访问方式、数据安全要求、并发人数和核心使用场景。

## 2. 硬件环境检查
1. CPU 型号和核心数：影响服务启动、数据处理和部分推理速度。
2. 内存大小：影响模型服务、网页服务和依赖程序能否稳定运行。
3. 显卡型号：影响是否支持 GPU 加速和推理性能。
4. 显存大小：决定可加载模型大小、上下文长度和并发能力。
5. 磁盘总容量：用于存放模型文件、依赖、镜像、日志和用户数据。
6. 磁盘剩余空间：防止模型下载、依赖安装和日志写入时空间不足。
7. 是否为 SSD：影响模型加载、服务启动和数据读写速度。
8. 供电稳定性：防止服务中断、数据损坏和部署失败。

## 3. 系统环境检查
1. 操作系统类型：确认是 Windows、Linux 还是 macOS，便于选择正确安装包。
2. 系统版本：部分软件对系统版本有最低要求。
3. 系统架构：确认是 x64、ARM 等架构，避免下载错误版本。
4. 是否支持目标软件：确认系统能否运行 Ollama、Docker、Open WebUI 等工具。
5. 是否支持虚拟化或 WSL：部分容器、Linux 环境或开发工具需要这些能力。
6. 是否有系统更新限制：避免系统更新后造成驱动、依赖或软件兼容问题。

## 4. 网络环境检查
1. 是否能访问官网：确认能否获取安装包和官方文档。
2. 是否能下载模型或镜像：模型和容器镜像通常体积较大，需要稳定网络。
3. 内网是否能访问服务：确认公司内部用户能否访问部署好的网页或 API。
4. 端口是否开放：确认服务端口未被占用，并且允许访问。
5. 防火墙是否拦截：避免服务启动后网页或接口无法访问。
6. 是否需要代理：部分环境下载模型或依赖时可能需要代理配置。

## 5. 权限环境检查
1. 是否有管理员权限：安装软件、启动服务、开放端口可能需要管理员权限。
2. 是否能安装软件：确认公司电脑或服务器允许安装目标工具。
3. 是否能打开终端：部署、测试和排查问题通常需要使用命令行。
4. 是否能修改配置文件：端口、路径、账号、服务地址等通常需要配置。
5. 是否能访问安装目录和数据目录：避免服务无法读取模型或写入日志。
6. 是否能管理账号权限：方便后续做用户分级、离职回收和数据安全控制。

## 6. 依赖环境检查
1. Python：常用于后端服务、AI 应用、数据处理和自动化脚本。
2. Node.js：常用于前端工程、Web 服务和工具链运行。
3. Docker：用于容器化部署，把应用和运行环境一起打包，减少环境差异。
4. Git：用于版本管理、代码拉取、文档管理和协作。
5. CUDA：用于 NVIDIA 显卡加速大模型推理，需要和显卡驱动匹配。
6. 显卡驱动：决定系统能否正确识别和使用显卡。
7. 数据库：部分系统需要 MySQL、PostgreSQL 或向量数据库存储数据。

## 7. 业务环境检查
1. 谁使用系统：确认是财务、采购、业务还是技术人员，便于设计功能和权限。
2. 有多少用户：评估并发压力、硬件配置和服务稳定性要求。
3. 核心业务流程是什么：确保部署后的系统能支持真实业务流程。
4. 旧数据在哪里：确认是否需要迁移、清洗、备份或导入数据。
5. 是否需要账号权限分级：保护客户资料、财务数据和内部知识。
6. 上线时间是否有限制：避免影响业务高峰期。
7. 失败后是否需要回滚：提前准备恢复方案，降低上线风险。

## 8. 可能风险
1. 硬件配置不足，导致模型无法加载或响应很慢。
2. 网络不稳定，导致模型、镜像或依赖下载失败。
3. 权限不足，导致无法安装软件、修改配置或启动服务。
4. 依赖版本不匹配，导致服务启动失败。
5. 端口被占用或防火墙拦截，导致网页或 API 无法访问。
6. 业务需求不清晰，导致部署完成后不能满足实际使用。
7. 没有备份和回滚方案，导致上线失败后难以恢复。

## 9. 面试表达
我在部署前会先做环境核验，主要从硬件、系统、网络、权限、依赖和业务六个方面检查。重点确认机器性能是否满足、系统版本是否兼容、网络和端口是否可用、权限是否足够、依赖是否完整，以及业务目标是否清晰，提前降低部署失败和返工风险。
