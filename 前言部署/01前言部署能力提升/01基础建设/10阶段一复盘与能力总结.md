## 2. 阶段一已学习内容

### 2.1 Markdown

Markdown 是一种轻量级文档格式，可以用简单符号表达标题、列表、表格、代码块、任务清单等结构。

它适合编写部署文档、学习笔记、SOP、测试报告和项目 README。

需要注意：Markdown 不是“AI 语言”，而是技术文档中常用的轻量化标记语言。

### 2.2 Obsidian

Obsidian 是本地化知识库工具，底层使用 Markdown 文件保存内容。

它可以通过双向链接、目录结构和标签，把系统部署、AI 大模型、GitHub、ERP/WMS、SOP 和项目记录串联起来，形成个人知识体系。

当前已在 Obsidian 中建立前沿部署学习目录，并将阶段一学习内容进行归档。

### 2.3 Git/GitHub

Git 是本地版本控制工具，用于记录部署文档、配置文件、脚本和项目代码的修改历史。

GitHub 是线上代码和文档托管平台，可以用于远程备份、团队协作和项目展示。

需要注意：Git 是版本控制工具，GitHub 本身是托管平台；GitHub 依赖 Git 的版本控制能力。

### 2.4 系统部署

系统部署不是简单安装软件，而是围绕客户或业务目标，把系统、模型、数据库、网络、权限、配置和文档准备好，并通过测试验证，让用户能够稳定使用。

完整部署流程包括：

1. 需求确认
2. 环境核验
3. 安装部署
4. 参数配置
5. 功能测试
6. 上线发布
7. 监控运维
8. 复盘文档

### 2.5 环境核验

环境核验是在部署前确认机器、系统、网络、权限、依赖和业务条件是否满足部署要求。

主要检查维度包括：

1. 硬件环境：CPU、内存、显卡、显存、磁盘等。
2. 系统环境：操作系统、系统版本、系统架构、兼容性等。
3. 网络环境：官网访问、模型下载、端口、防火墙、代理等。
4. 权限环境：管理员权限、安装权限、配置文件权限等。
5. 依赖环境：Python、Docker、Git、CUDA、数据库等。
6. 业务环境：使用对象、并发人数、数据安全、权限分级、上线时间等。

### 2.6 AI 基础概念

已学习以下 AI 大模型基础概念：

1. LLM：大语言模型。
2. Token：模型处理文本的基本单位，和资源消耗有关。
3. Context Window：上下文窗口，表示模型一次能处理和记住的文本范围。
4. Embedding：向量化，用于把文本转换成数字向量，是 RAG 检索的基础。
5. 参数规模：例如 7B、14B、32B，表示模型参数量，通常参数越大资源要求越高。

需要注意：Embedding 不是“让 AI 直接理解全部内容”，而是把文本转成向量，方便系统做相似度检索。

## 6. 遇到的问题和解决方式

|问题|原因|解决方式|
|---|---|---|
|Git 提交提示身份未知|未配置 Git 用户名和邮箱|配置 `git config user.name` 和 `git config user.email`|
|命令 `1s` 无法识别|将 `ls` 的字母 `l` 输成数字 `1`|改为正确命令 `ls`|
|模型下载可能受网络影响|网络环境或下载源限制|检查网络，必要时更换网络或使用可访问的下载方式|

## 8. 面试问答：如何做部署前环境核验

```
部署前我会先根据业务目标确认部署对象和使用场景，然后从硬件、系统、网络、权限、依赖和业务六个方面做环境核验。重点检查 CPU、内存、磁盘、系统版本、网络访问、端口、防火墙、安装权限、Docker/Python/Git 等依赖，以及用户数量、数据安全和上线时间。环境核验完成后，再进行安装、配置、测试和交付，避免部署过程中反复返工。
```
