## 课程目标

第24课学习 API 限流，也就是限制单位时间内允许访问的次数。

本课解决的问题：

- 防止某个客户端连续大量调用模型。
- 避免模型资源被单个请求方占满。
- 在请求过多时返回明确的错误状态。

## 一、限流的基本逻辑

本课使用的课堂规则是：

~~~
同一个客户端 10 秒内最多请求 3 次
超过次数 → 返回 429
等待时间结束 → 恢复访问
~~~

完整流程：

~~~
请求进入
    ↓
身份验证
    ↓
读取客户端 IP
    ↓
检查最近 10 秒的请求记录
    ↓
没有超过 3 次 → 继续调用 Ollama
超过 3 次 → 返回 429
~~~

## 二、通过什么实现

主要使用：

- time.monotonic：记录稳定的时间间隔。
- 字典 request_records：保存每个客户端的请求记录。
- check_rate_limit：检查请求是否超过限制。
- HTTP 状态码 429：表示请求过于频繁。

核心配置：

~~~python
RATE_LIMIT = 3
WINDOW_SECONDS = 10
request_records: dict[str, list[float]] = {}
~~~

- RATE_LIMIT：时间窗口内允许的最大请求次数。
- WINDOW_SECONDS：时间窗口长度。
- request_records：保存客户端 IP 和请求时间。

## 三、限流函数

~~~python
def check_rate_limit(client_id: str):
    now = time.monotonic()
    timestamps = request_records.get(client_id, [])

    timestamps = [
        timestamp
        for timestamp in timestamps
        if now - timestamp < WINDOW_SECONDS
    ]

    if len(timestamps) >= RATE_LIMIT:
        raise HTTPException(
            status_code=429,
            detail="请求过于频繁，请稍后再试"
        )

    timestamps.append(now)
    request_records[client_id] = timestamps
~~~

这段代码先删除时间窗口之外的旧记录，再计算最近 10 秒内还有多少条记录。如果达到 3 次，就拒绝本次请求；没有达到，就把本次请求时间保存下来。

## 四、在聊天接口中调用限流

~~~python
client_ip = (
    client_request.client.host
    if client_request.client
    else "unknown"
)

check_rate_limit(client_ip)
~~~

这表示以客户端 IP 作为限流依据。同一个 IP 的请求会共享同一个计数。

限流应该放在 API Key 验证之后、调用 Ollama 之前：

~~~text
先鉴权 → 再限流 → 最后调用模型
~~~

这样错误的 API Key 不会占用模型调用次数。

~~~

- 第 4 次返回 429 Too Many Requests。
- 等待 10 秒后，再次请求可以恢复为 200。

## 七、验证结果含义

~~~text
200 → 请求正常处理
429 → 请求次数超过限制
等待后 200 → 时间窗口已经恢复
~~~

实际终端中出现过：

~~~text
POST /chat -> 200
POST /chat -> 429 Too Many Requests
POST /chat -> 200
~~~

这证明限流逻辑已经生效。

## 八、本课的限制

本课使用的是 Python 内存中的字典，因此：

- 服务重启后记录会清零。
- 多个 FastAPI 进程之间不会共享记录。
- 不能用于多服务器生产环境。

正式系统通常使用 Redis 等独立存储保存限流记录，并配合专业限流组件。
