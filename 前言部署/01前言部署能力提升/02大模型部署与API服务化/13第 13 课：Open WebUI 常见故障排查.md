## 目标
学会判断 Open WebUI 出问题时，应该先查哪里、后查哪里
**需要达成的成功**
你能根据不同现象，说出对应的排查顺序

## 一、先记住排查总原则
先看现象，再分层排查
**Open WebUI 常见问题分 4 类：**
1. 页面打不开
2. 页面能打开，但没有模型
3. 有模型，但回答失败
4. 回答很慢或卡住

## 问题1**页面打不开**：
访问 http://localhost:3000 打不开
优先怀疑：Docker / Open WebUI / 3000端口
排查顺序
1. Docker Desktop 是否启动
2. open-webui 容器是否运行
3. open-webui 容器是否 Healthy
4. 3000 端口是否映射成功
5. 浏览器地址是否写错
6. open-webui 日志是否报错
常用命令：docker ps
重点看三列：
STATUS：是不是 Up / Healthy
PORTS：有没有 0.0.0.0:3000->8080
NAMES：是不是 open-webui
总结：页面打不开，先查 Open WebUI 服务和 3000 端口。

## 问题 2：页面能打开，但没有模型
Open WebUI 页面正常，但模型列表为空
优先怀疑：Ollama / 模型列表 / 11434端口 / WebUI连接配置
排查的顺序：
1. Ollama 是否安装并运行
2. ollama list 是否有模型
3. 用 ollama run 模型名 验证模型能否被正常加载和回答
4. 11434 端口是否可访问
5. Open WebUI 里 Ollama 地址是否配置正确
常见命令：
ollama --version
ollama list
ollama run qwen2.5:3b
总结：页面有了但没模型，查 Ollama 和模型连接

## 问题 3：有模型，但回答失败
模型列表能看到模型，但发送问题后报错或没有回答
优先怀疑：模型加载失败 / 资源不足 / Ollama调用失败
排查顺序：
1. 在命令行里直接运行模型，看是否能回答
2. 检查模型是否下载完整
3. 检查内存、显存、CPU占用
4. 查看 Open WebUI 日志
5. 查看 Ollama 是否报错
总结：有模型但不能回答，查模型运行能力和系统资源

## 问题 4：回答很慢或卡住
模型能回答，但非常慢，或者一直转圈
优先怀疑：模型太大 / 电脑资源不足 / 同时运行服务太多
排查顺序：
1. 换一个更小的模型测试
2. 查看任务管理器里的 CPU、内存、显存
3. 关闭不必要的容器或软件
4. 检查是否同时多人访问
5. 判断是否需要更高配置机器
总结：慢不一定是坏，多数是资源不足。

**排查口诀：**
页面打不开，查 WebUI。
页面没模型，查 Ollama。
模型不回答，查运行。
回答太慢，查资源。

不要凭着感觉修，要根据现象定位问题
Windows电脑 = 宿主机
Docker里的Open WebUI = 容器
Ollama = 可能跑在宿主机上

容器里的 localhost，不等于 Windows 的 localhost
