# 总结
# Markdown 基础与部署文档写法

## 1. 当前课程定位

本课程属于阶段一：基础建设。

该阶段中，Markdown 是后续学习记录、部署方案、环境核验表、SOP、测试报告和复盘文档的基础格式。后续 Obsidian 知识库、GitHub 文档管理和项目作品集都会大量使用 Markdown。

## 2. 对 Markdown 的理解

Markdown 是一种轻量级文档格式，可以用简单符号表达标题、列表、表格、代码块、任务清单等结构。

它适合技术文档的原因是结构清晰、容易维护、适合版本管理，也方便在 Obsidian、GitHub、VS Code 等工具中使用。

## 3. 常用 Markdown 语法

### 3.1 标题

```markdown
# 一级标题
## 二级标题
### 三级标题
```

标题用于组织文档结构，让读者快速知道每一部分内容。

### 3.2 加粗

```markdown
**重要内容**
```

加粗适合强调风险、结论、注意事项。

### 3.3 无序列表

```markdown
- CPU
- 内存
- 磁盘
- 网络
```

无序列表适合列出检查项、工具清单和注意事项。

### 3.4 有序列表

```markdown
1. 下载软件
2. 安装服务
3. 修改配置
4. 测试访问
```

有序列表适合描述部署步骤、测试步骤和排查顺序。

### 3.5 行内代码

```markdown
`ollama run qwen2`
```

行内代码适合记录命令、端口、文件名、配置项。

### 3.6 代码块

````markdown
```bash
ollama list
ollama run qwen2
```
````

代码块适合记录多行命令、配置文件和脚本。

### 3.7 表格

```markdown
| 检查项 | 要求 | 结果 |
| --- | --- | --- |
| 内存 | 16GB 以上 | 待检查 |
| 磁盘 | 20GB 以上 | 待检查 |
| 网络 | 能访问下载源 | 待检查 |
```

表格适合环境核验、风险清单和测试记录。

### 3.8 任务清单

```markdown
- [ ] 检查系统版本
- [ ] 检查内存
- [ ] 安装 Ollama
- [ ] 测试模型运行
```

任务清单适合做部署 Checklist。

## 4. 为什么部署文档适合用 Markdown

1. 结构清晰：可以通过标题、列表和表格清楚表达部署流程。
2. 容易维护：修改文档时不需要复杂排版，适合长期更新。
3. 适合版本管理：可以放到 GitHub 中记录每次修改。
4. 方便工具识别：Obsidian、GitHub、VS Code 和 AI 工具都能很好识别 Markdown。
5. 便于复用：同一套文档结构可以复用于 Ollama、Open WebUI、Odoo、RAG 等不同项目。

## 5. 标准部署文档结构

一份标准部署文档建议包含以下部分：

1. 部署目标：说明要部署什么、给谁用、解决什么问题。
2. 环境要求：说明硬件、系统、网络、权限、依赖和业务条件。
3. 部署步骤：记录从安装到启动的完整过程。
4. 参数配置：记录端口、路径、账号、服务地址、模型名称等关键配置。
5. 测试验证：说明如何判断部署成功，例如网页能打开、接口能返回、核心功能能运行。
6. 常见问题：记录部署中可能出现的错误、原因和解决办法。
7. 交付说明：说明访问方式、启动方式、维护方式和注意事项。
8. 复盘总结：记录部署经验、问题改进和后续优化方向。

## 6. 普通学习笔记和部署文档的区别

普通学习笔记主要给自己看，内容可以比较自由，重点是记录个人理解。

部署文档主要给别人复用，要求结构清晰、步骤明确、配置可查、结果可验证、问题可追踪。

核心区别如下：

1. 普通学习笔记偏个人理解，部署文档偏交付复用。
2. 普通学习笔记可以较零散，部署文档必须有固定结构。
3. 普通学习笔记主要帮助自己回忆，部署文档要让别人照着也能操作。
4. 普通学习笔记不一定有验证标准，部署文档必须说明如何判断部署成功。
