## 课程目标

第25课学习 API 日志和运行状态监控。

本课需要知道：

- 有没有请求进入。
- 请求访问了哪个路径。
- 请求返回了什么状态码。
- 请求处理花费了多长时间。
- 当前成功请求和失败请求各有多少。

## 一、日志和指标的区别

日志是对单次请求的详细记录：

~~~text
GET /metrics -> 200，耗时 7.38 ms
~~~

指标是对整体运行情况的统计：

~~~json
{
  "total_requests": 10,
  "success_requests": 8,
  "error_requests": 2
}
~~~

可以理解为：

- 日志：记录每一件事情。
- 指标：统计整体情况。

## 二、通过什么实现

本课使用 Python 自带的 logging 模块和 FastAPI 中间件。

主要组件：

- logging：输出运行日志。
- middleware：让每个请求自动经过统一处理。
- time.perf_counter：计算请求耗时。
- total_requests：统计总请求数。
- success_requests：统计成功数。
- error_requests：统计错误数。
- GET /metrics：通过接口查看统计结果。

## 三、配置日志和统计变量

~~~python
import logging
import time

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("fastapi-monitor")

total_requests = 0
success_requests = 0
error_requests = 0
~~~

这段代码创建日志记录器，并准备三个内存计数器。

## 四、监控中间件

~~~python
@app.middleware("http")
async def monitor_requests(request: Request, call_next):
    global total_requests, success_requests, error_requests

    start_time = time.perf_counter()
    total_requests += 1

    try:
        response = await call_next(request)
    except Exception:
        error_requests += 1
        raise

    duration = (time.perf_counter() - start_time) * 1000

    if response.status_code < 400:
        success_requests += 1
    else:
        error_requests += 1

    logger.info(
        "%s %s -> %s，耗时 %.2f ms",
        request.method,
        request.url.path,
        response.status_code,
        duration
    )

    return response
~~~

### 代码工作过程

1. 请求进入中间件。
2. 记录开始时间。
3. 总请求数加一。
4. 把请求交给具体接口处理。
5. 得到接口返回的状态码。
6. 计算处理耗时。
7. 根据状态码增加成功数或错误数。
8. 把请求方法、路径、状态码和耗时写入终端。
9. 把原响应返回给浏览器。

## 五、增加 metrics 接口

~~~python
@app.get("/metrics")
def get_metrics():
    return {
        "total_requests": total_requests,
        "success_requests": success_requests,
        "error_requests": error_requests
    }
~~~

访问：

~~~text
http://127.0.0.1:8000/metrics
~~~

可以查看当前服务的内存统计。

## 七、验证结果

访问 metrics 后，网页会显示类似：

~~~json
{
  "total_requests": 1,
  "success_requests": 0,
  "error_requests": 0
}
~~~

刚启动服务后，统计从零开始。第一次访问 metrics 时，网页读取的是中间件更新完成之前的统计快照，因此 success_requests 可能暂时还是 0。

终端会随后显示：

~~~text
GET /metrics -> 200，耗时 7.38 ms
~~~

这才说明本次 metrics 请求最终成功。

## 八、终端日志的实际含义

~~~text
POST /chat -> 200
~~~

表示聊天请求成功。

~~~text
POST /chat -> 429 Too Many Requests
~~~

表示请求触发了第24课的限流规则。

~~~text
GET /favicon.ico -> 404
~~~

表示浏览器请求网页图标，但项目没有配置图标。这个错误不影响 API。

## 九、为什么重启后统计会清零

本课的计数器保存在 Python 进程内存中：

~~~python
total_requests = 0
~~~

服务停止后，内存被释放；服务重新启动后，变量又从零开始。因此本课的统计适合学习和演示，不适合长期生产监控。

正式环境通常会使用 Prometheus、Grafana、日志平台或数据库保存历史数据。
